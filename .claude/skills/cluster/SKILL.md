# Clustering Schema Lifecycle

A clustering schema is an **immutable artifact on disk** containing both cluster
routes and expert routes (ranks 1/2/3) for a session, across **all 4 fixed
layer windows**. Output buckets are baked in at build time as `ground_truth`
(friend / foe / unknown). Filters (`steps`, `last_occurrence_only`, `max_probes`)
and the UMAP/clustering params are also baked in. Schemas live at:

```
$ROOT/data/lake/<session_id>/clusterings/<schema_name>/
├── meta.json
├── windows/
│   ├── w0/
│   │   ├── cluster/
│   │   │   └── t_<a>_<b>.json                    # one file per transition
│   │   └── expert/
│   │       └── t_<a>_<b>__rank{1,2,3}.json       # all three ranks
│   ├── w1/  (same shape)
│   ├── w2/  (same shape)
│   └── w3/  (same shape)
├── centroids.json
├── probe_assignments.json
├── trajectory_points.json                # 3D UMAP for the trajectory plot
├── reports/                              # per-window synthesis (auto-generated by /analyze chain)
├── element_descriptions.json             # cluster/route labels (auto-generated)
└── _archive/<schema>_<ts>/               # archived schemas
```

**Auto-analyze**: OP-1 builds the schema, then automatically chains to `/analyze`
for each of the 4 windows. The user gets a fully-analyzed schema in one
invocation. Pass `skip-analyze` after the curl args to opt out for fast iteration.

**Terminology:**
- A **window** is a 6-layer range. There are 4 fixed windows: `w0=[0,5]`,
  `w1=[5,11]`, `w2=[11,17]`, `w3=[17,23]`. Each schema covers all 4.
- A **transition** is a 2-layer pair (e.g. `t_22_23`) showing before-cluster +
  after-cluster + the routes between them. There are 6 transitions per window.

**Schemas are atomic and immutable.** A build either succeeds entirely or
leaves no trace; the only legal evolutions are "build a fresh schema" or
"archive + rebuild." There is no per-window or per-transition rebuild — the
unit of work is the whole schema.

The frontend has zero schema-management UI. Everything is curl from this skill.

## Prerequisites

1. `/server` OP-1 to confirm the backend is up and the model is loaded.
2. The session's probe captures must already be on disk (`tokens.parquet`,
   `residual_stream/*.parquet`).

## Default clustering config (single source of truth)

The backend's `ClusteringConfig` is a **flat** Pydantic model. The fields below
match `backend/src/api/schemas.py:ClusteringConfig` exactly — do not nest them.

```yaml
defaults_common:
  embedding_source:    residual_stream     # or "expert_output"
  reduction_method:    umap                # or "pca"
  reduction_dimensions: 6                  # UMAP n_components
  n_neighbors:         15                  # UMAP n_neighbors
  clustering_method:   hierarchical        # or "kmeans" / "dbscan"
  default_k:           6                   # k clusters per layer (expand into layer_cluster_counts at build time)

filter_defaults:
  last_occurrence_only: true
  max_probes:           null

session_kind:
  probe:    { steps_default: null }        # sentence-set runs — token rows have step=None, so omit the filter (passing [0] returns zero probes)
  agent:    { steps_default: [1] }         # post-examine tick (turn_id=1 by convention)

schema_name_convention:
  # Concise pattern (per existing bus_stop_friend_foe_k6_n15):
  #   <study-and-axes>_k<k>_n<n>[_<sweep-suffix>]
  # OP-1 requires explicit save_as. Sweep callers append a short per-axis suffix:
  #   max_probes  → _max<N>     (e.g. _max25)
  #   steps       → _step<vals> (e.g. _step01)
  #   reduction_dimensions → _d<N> (e.g. _d3)
  #   n_neighbors → _n<N>       (e.g. _n5)
  example_base: bus_stop_friend_foe
  example_full: bus_stop_friend_foe_k6_n15
```

This block is the only place to update defaults. The `/probe` and `/agent`
post-run hooks read from here when proposing a default schema build.

`layer_cluster_counts` is per-layer. Since every schema now covers all 4 fixed
windows (layers 0–23), supply one entry for each layer:
`{"0":6,"1":6,...,"23":6}`. With `default_k=6` that's 24 identical entries.

## Operations

### OP-1: Build a schema (atomic)

Builds the entire schema in **one call**: cluster routes plus expert routes
(ranks 1/2/3) for every transition in every one of the 4 fixed windows. Either
succeeds completely or leaves nothing on disk.

**`save_as` is required and must NOT already exist.** If it does, the endpoint
returns 409 — pick a new name or delete/archive the old one.

Replace `SESSION_ID` and `SAVE_AS`. Defaults below match the YAML block above.

```bash
ROOT=$(git rev-parse --show-toplevel) && PY="$ROOT/.venv/bin/python" && \
SID=SESSION_ID && SAVE=SAVE_AS && \
curl -s -X POST http://localhost:8000/api/experiments/build-schema \
  -H "Content-Type: application/json" \
  -d '{
    "session_id":"'"$SID"'",
    "save_as":"'"$SAVE"'",
    "clustering_config":{
      "embedding_source":"residual_stream",
      "reduction_method":"umap",
      "reduction_dimensions":6,
      "n_neighbors":15,
      "clustering_method":"hierarchical",
      "layer_cluster_counts":{"0":6,"1":6,"2":6,"3":6,"4":6,"5":6,"6":6,"7":6,"8":6,"9":6,"10":6,"11":6,"12":6,"13":6,"14":6,"15":6,"16":6,"17":6,"18":6,"19":6,"20":6,"21":6,"22":6,"23":6}
    },
    "steps":[1],
    "last_occurrence_only":true,
    "top_n_routes":20,
    "output_grouping_axes":["ground_truth"]
  }' | $PY -m json.tool
```

**Sentence-experiment sessions:** omit `steps` entirely (or pass `null`). Token records have `transition_step=None` and the `step` filter excludes everything if you pass `[0]` or `[1]`. The example above uses `[1]` because it assumes an agent session — adapt accordingly.

**`output_grouping_axes` — pick the right value per session type**:

| Session type | Recommended value | Right-column buckets |
|---|---|---|
| **Agent + want NPC truth** (default for bus_stop) | `["ground_truth"]` | foe / friend |
| **Agent + want agent action** | `["action_type"]` or omit | enemy / friend / neutral (post-enrichment) |
| **Sentence (post-categorize)** | omit | composite from `record.output_category` (engagement_fictional / etc.) |
| **Sentence (NOT yet categorized)** | omit | empty right column (acceptable; rebuild after /categorize) |

⚠️ **Footgun**: rebuilding bus_stop schemas WITHOUT explicitly passing `output_grouping_axes=["ground_truth"]` will silently switch their right column from foe/friend to enemy/friend/neutral. Always specify for agent sessions.

Expected: `{"schema": "...", "windows": 4, "transitions": 24, "ranks": 3, "sample_size": ..., "total_probes": ...}`.

**Verification block (run after OP-1 returns):**

```bash
ROOT=$(git rev-parse --show-toplevel) && SID=SESSION_ID && SAVE=SAVE_AS && \
echo "=== Schema root ===" && ls -1 "$ROOT/data/lake/$SID/clusterings/$SAVE/" && \
echo "=== Windows ===" && ls -1 "$ROOT/data/lake/$SID/clusterings/$SAVE/windows/" && \
echo "=== w3/cluster ===" && ls -1 "$ROOT/data/lake/$SID/clusterings/$SAVE/windows/w3/cluster/" && \
echo "=== w3/expert ===" && ls -1 "$ROOT/data/lake/$SID/clusterings/$SAVE/windows/w3/expert/" && \
echo "=== meta.json ===" && cat "$ROOT/data/lake/$SID/clusterings/$SAVE/meta.json"
```

Expected: `meta.json` (with `params`, `expert_routes_params`, and `windows`
blocks), `centroids.json`, `probe_assignments.json`, `trajectory_points.json`,
and a `windows/{w0,w1,w2,w3}/{cluster,expert}/` tree containing
`t_<a>_<b>.json` and `t_<a>_<b>__rank{1,2,3}.json` files. **No** `__out_*` variants —
output buckets are baked at build time as `ground_truth` (friend/foe/unknown).

### OP-1A: Auto-analyze chain (run AFTER OP-1)

After OP-1 returns 200, immediately run `/analyze` for each of the 4 windows.
This populates `reports/w_<a>_<b>.md` per-window synthesis files and
accumulates per-cluster/route descriptions in `element_descriptions.json`.

The reviewer/analyst then opens MUDApp, picks the schema, and immediately sees
populated reports + click-to-inspect cards — no extra step required.

```
/analyze {SID} schema {SAVE} window 0-5
/analyze {SID} schema {SAVE} window 5-11
/analyze {SID} schema {SAVE} window 11-17
/analyze {SID} schema {SAVE} window 17-23
```

Each window pass takes ~5-10 min of Claude reasoning. Run sequentially or
batch in parallel via the Agent tool if you want speed.

To **skip** the auto-analyze (e.g. parameter exploration with no need for
written reports), pass the `skip-analyze` keyword after OP-1 returns and
exit without invoking /analyze.

### OP-2: Sweep — call OP-1 in a shell loop with suffixed names

A sweep is just OP-1 N times with a per-axis suffix on `save_as`. No formal
sweep machinery — write a `for` loop. For multi-axis sweeps, nest loops; the
suffix concatenates per axis (e.g. `_max25_step01`).

Example sweep over `max_probes ∈ {25, 50, 100}`:

```bash
ROOT=$(git rev-parse --show-toplevel) && PY="$ROOT/.venv/bin/python" && \
SID=SESSION_ID && BASE=BASE_NAME && \
for N in 25 50 100; do \
  SAVE="${BASE}_max${N}" && \
  echo "=== ${SAVE} ===" && \
  curl -s -X POST http://localhost:8000/api/experiments/build-schema \
    -H "Content-Type: application/json" \
    -d '{
      "session_id":"'"$SID"'",
      "save_as":"'"$SAVE"'",
      "clustering_config":{
        "embedding_source":"residual_stream",
        "reduction_method":"umap",
        "reduction_dimensions":6,
        "n_neighbors":15,
        "clustering_method":"hierarchical",
        "layer_cluster_counts":{"0":6,"1":6,"2":6,"3":6,"4":6,"5":6,"6":6,"7":6,"8":6,"9":6,"10":6,"11":6,"12":6,"13":6,"14":6,"15":6,"16":6,"17":6,"18":6,"19":6,"20":6,"21":6,"22":6,"23":6}
      },
      "steps":[1],
      "last_occurrence_only":true,
      "max_probes":'"$N"',
      "top_n_routes":20
    }' | $PY -m json.tool ; \
done
```

Each schema is fully usable as soon as its build call returns. The loop
continues even if a single iteration 5xx's — read the per-iteration JSON to
spot failures. If you want strict halt-on-fail, wrap the curl in `|| break`.

**Wall-time per schema:** ~12 min full-sample build (UMAP fit dominates the
per-transition cost; 24 transitions × ~30 s each). Sub-sampled builds with
small `max_probes` are similar — UMAP cost is roughly constant, not linear in
sample size. Plus ~5-10 min per window for the auto-analyze chain (40 min
total for all 4 windows). For a sweep over N schemas, multiply by N.

### OP-3: List schemas for a session

```bash
ROOT=$(git rev-parse --show-toplevel) && PY="$ROOT/.venv/bin/python" && \
SID=SESSION_ID && \
curl -s http://localhost:8000/api/probes/sessions/$SID/clusterings | $PY -m json.tool
```

Returns a `clusterings` array of `meta.json` blobs (name, created_at, params,
filters, sample_size, expert_routes_params).

### OP-4: Archive a schema

Moves the schema directory to `clusterings/_archive/<name>_<ts>/`. The schema
disappears from the UI's list but is recoverable by `mv`-ing back manually.

```bash
ROOT=$(git rev-parse --show-toplevel) && PY="$ROOT/.venv/bin/python" && \
SID=SESSION_ID && SAVE=SCHEMA_NAME && \
curl -s -X POST http://localhost:8000/api/probes/sessions/$SID/clusterings/$SAVE/archive | $PY -m json.tool
```

Manual restore:

```bash
ROOT=$(git rev-parse --show-toplevel) && SID=SESSION_ID && \
mv "$ROOT/data/lake/$SID/clusterings/_archive/<archived_name>" "$ROOT/data/lake/$SID/clusterings/<original_name>"
```

### OP-5: Delete a schema

Permanent. Returns 409 if `reports/` or `element_descriptions.json` exist
(those represent invested analysis time). Override with `?force=true`.

```bash
ROOT=$(git rev-parse --show-toplevel) && PY="$ROOT/.venv/bin/python" && \
SID=SESSION_ID && SAVE=SCHEMA_NAME && \
curl -s -X DELETE "http://localhost:8000/api/probes/sessions/$SID/clusterings/$SAVE" | $PY -m json.tool
```

Force delete (only when you're certain):

```bash
ROOT=$(git rev-parse --show-toplevel) && PY="$ROOT/.venv/bin/python" && \
SID=SESSION_ID && SAVE=SCHEMA_NAME && \
curl -s -X DELETE "http://localhost:8000/api/probes/sessions/$SID/clusterings/$SAVE?force=true" | $PY -m json.tool
```

## Common workflows

### Build the default schema after a session completes

1. `/server` OP-1 to confirm backend ready.
2. OP-1 here with `save_as=<study>_k6_n15`. The build covers all 4 fixed
   windows × 6 transitions × {cluster + expert ranks 1/2/3} in one call.
3. Verification block above to confirm files on disk.
4. Open MUDApp, pick the session, pick the new schema in the Toolbar — every
   window in the dropdown loads instantly from cache.

### Subsample sweep for paper

Run OP-2 with `max_probes ∈ {25, 50, 100, 150, 200, 300, 400}`, `k=6`. After
the sweep, OP-3 lists all 7 schemas; each one can be analyzed separately via
`/analyze`.

### Recover from a misconfigured build

OP-4 (archive) preserves the bad schema for forensic inspection. Then OP-1
with the corrected params using a fresh `save_as`. If you're sure the bad
one is worthless, OP-5 (delete) skips the archive step.

## Troubleshooting

### 409 — Schema already exists

`save_as` collides with an existing schema directory. Pick a different name,
or archive (OP-4) / delete (OP-5) the existing one first.

### 409 on delete — `schema_has_invested_data`

The schema has reports or element_descriptions. Default refuses delete to
prevent accidental loss of analysis work. Pass `?force=true` to override.

### 404 — Schema not found

Check OP-3 to see what schemas exist. The `_archive/` directory is excluded
from the listing.

### 404 on trajectory endpoint

The schema was built before `trajectory_points.json` existed. Per the
no-migration policy: archive (OP-4) and rebuild (OP-1) — don't try to patch
the missing file.

### 5xx mid-build with no schema directory afterwards

Expected behaviour. The build endpoint writes everything into a temp dir
first and only `os.rename`s into place once all 4 windows × 6 transitions ×
3 ranks have been written. Any exception triggers `shutil.rmtree` of the temp
dir and re-raises as a 500. Read the response body for the failing
window/transition/rank.

## Important rules

- **`save_as` is required.** No auto-generated names.
- **Schemas are atomic and immutable.** A build either succeeds in full or
  writes nothing; once written, the only legal evolutions are archive or
  delete and rebuild under a new name.
- **A schema covers all 4 fixed windows.** OP-1 has no `windows` parameter —
  the windows are hardcoded server-side (see `backend/src/api/layer_windows.py`).
- **Cluster + expert routes are one schema.** OP-1 always builds both.
- **Never delete without checking for reports.** The 409 guard catches the
  common mistake; only force-delete when you're sure nothing of value lives
  in the schema.
- **Frontend is a pure viewer.** Schema lifecycle lives here, not in the UI.
